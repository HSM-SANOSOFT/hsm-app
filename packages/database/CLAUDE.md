# CLAUDE.md — `@hsm/database`

Global `DatabaseModule` plus a CLI for generating TypeORM entities/schemas from Oracle.

## What's exported

```ts
import { DatabaseModule } from '@hsm/database';
```

`DatabaseModule` is `@Global()`, so importing it once in `MainModule` / `WorkerModule` is enough — repositories are available everywhere via `@InjectRepository`.

It also registers `TypeOrmExceptionFilter` as an `APP_FILTER`, so TypeORM errors don't need per-controller catches.

## Two data sources

Defined in `src/sources/database-source.enum.ts`:

```ts
export enum DatabasesEnum {
  HsmDbPostgres = 'hsm-db-postgres',
  HsmDbOracle   = 'hsm-db-oracle',
}
```

Inject by name:

```ts
@InjectRepository(Patient, DatabasesEnum.HsmDbPostgres)
private readonly patients: Repository<Patient>;
```

`synchronize: true` is **dev-only**. Never enable in prod — Oracle is the system of record.

## Adding a Postgres entity

### 1. Create the entity file

`src/entities/modules/<domain>/<entity>.entity.ts`

Use `DatabasePostgresSchemasEnum` for the schema name. Import it from `@hsm/database/sources/postgres` (schemas only — see circular dep note below).

```ts
import { DatabasePostgresSchemasEnum } from '@hsm/database/sources/postgres';
import { Entity, PrimaryGeneratedColumn, Column } from 'typeorm';

@Entity({ name: 'my_table', schema: DatabasePostgresSchemasEnum.CORE })
export class MyEntity {
  @PrimaryGeneratedColumn('uuid') id: string;
  // ...
}
```

### 2. Export from the domain barrel

Add to `src/entities/modules/<domain>/index.ts`:

```ts
export * from './my.entity';
```

### 3. Ensure the domain barrel is on the parent index

Check the barrel chain up to `src/entities/modules/core/index.ts` (or `security/index.ts`). If the domain folder is new, add it:

```ts
// src/entities/modules/core/index.ts
export * from './my-new-domain';
```

**This step was the root cause of the template entities bug.** `template/` existed but was missing from `core/index.ts`, so the entities never reached `@hsm/database/entities` and were silently absent from `databasePostgresEntities`.

### 4. Import in app code via root path only

```ts
// correct — always use the root path
import { MyEntity } from '@hsm/database/entities';

// NEVER use deep subpaths — causes circular dep (see below)
// import { MyEntity } from '@hsm/database/entities/modules/core/my-domain';
```

### 5. Verify startup

Run `start:dev` on both apps after adding an entity. Build success means nothing — DI failures from missing entity registration only surface at runtime.

## Entity import rules (circular dependency hazard)

Entity files import `DatabasePostgresSchemasEnum` from `@hsm/database/sources/postgres`. That index must only re-export `./database-postgres.schemas` — **never `./database-postgres.entities`**.

Why: `database-postgres.entities.ts` imports entity barrels → entity files import `sources/postgres` → if `sources/postgres` re-exports `database-postgres.entities`, the cycle is complete. In webpack's CommonJS module system this causes the entity barrel to be evaluated with empty exports at the moment `databasePostgresEntities` is computed — `Object.values(entityBarrel) === []` — so those entities are silently dropped from `forFeature` registration. Result: `UnknownDependenciesException` at startup despite correct `@Entity()` decorators and a successful build.

Additionally, deep subpath imports (e.g. `@hsm/database/entities/modules/core/template`) in app-code services cause the entity barrel to load before `database-postgres.entities` can finish evaluating, triggering the same empty-array outcome for that barrel group.

**`sources/postgres/index.ts` must stay as-is:**

```ts
export * from './database-postgres.schemas';
// never add: export * from './database-postgres.entities';
```

**Symptom:** an entity group is fully absent from `databasePostgresEntities` at runtime. Debug: add `console.log(Object.values(templateEntities))` in `database-postgres.entities.ts` — empty array confirms the circular dep fired.

## Commands

```bash
pnpm --filter @hsm/database build

# Generator CLI — run inside the hsm-app-be-api container (TypeScript drivers are there)
pnpm --filter @hsm/database generator:entity -- \
  --driver=oracle --host=... --user=... --pass=... --db=... --table=... --schema=... --save
pnpm --filter @hsm/database generator:schema
```

The generator imports from `src/generator/` and writes templates to `entities/autogenerated/oracle/`. Drivers bundled in `package.json`: `oracledb`, `pg`, `mysql2`, `mssql`, `sqlite3`.

## Postgres migrations (KTD9 — tooling now, prod activation deferred)

The Postgres source has a TypeORM migration runner + CLI. **Dev still uses
`synchronize`** (`onModuleInit`, `ENVIRONMENT==='dev'`); the non-dev migration
runner exists but stays **inert** unless `DB_POSTGRES_RUN_MIGRATIONS=true`. The
clinical spine ships dev-only; the first prod-bound module flips the flag to deliver
schema via migrations instead of `synchronize`.

```bash
# CLI data source: src/sources/postgres/database-postgres.datasource.ts
# (synchronize hard-OFF so migrations generate against a CLEAN, non-synced DB).

# Generate the baseline / a new migration after entities change (needs a live DB):
pnpm --filter @hsm/database migration:generate   # writes src/migrations/Clinical<ts>.ts
pnpm --filter @hsm/database migration:run        # apply pending migrations
pnpm --filter @hsm/database migration:revert
pnpm --filter @hsm/database migration:show
```

**Drift guard:** dev `synchronize` and migration-generated schema can silently
diverge (the classic TypeORM trap). ALWAYS `migration:generate` against a clean
(non-synchronized) DB, and commit the result. CI (`.github/workflows/migration-
drift.yml`) runs `migration:run` on a fresh DB then asserts `migration:generate`
produces **nothing** — a non-empty diff means entities changed without a matching
migration and fails the build. Migrations table: `_clinical_migrations`.

## Don't

- Don't open new TypeORM data sources from app code — extend `sources/` instead.
- Don't put domain logic in this package; it owns connections, entities, and the generator only.
- Don't hand-edit files under `entities/autogenerated/`.
