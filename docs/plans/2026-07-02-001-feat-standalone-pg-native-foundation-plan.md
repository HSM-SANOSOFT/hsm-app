# feat: Standalone pg-native foundation — disconnect Oracle & land the release train

## Summary

Make `@hsm/api` and `@hsm/worker` boot **without Oracle** by **unwiring the eager Oracle
datasource from app boot** (single seam: `database-sources.module.ts`) and dropping the
`DB_ORACLE_*` config requirement — **keeping every Oracle file on disk** for the eventual
cutoff migration. (Redis + MinIO remain boot dependencies; only Oracle is removed.) Add a CI regression that proves the API boots with no Oracle reachable,
then **land and activate the already-built release train** (development / release / main,
PR-gated, alpha/beta/stable tags, rulesets). This is the enabling foundation; per-cluster
greenfield module rebuilds are separate future plans.

---

## Problem Frame

The new app currently **fails to boot when Oracle is unreachable** (`ORA-12170`) because
`DatabaseOracleModule` registers an **eager** `TypeOrmModule.forRoot` that dials Oracle at
module init (`packages/database/src/sources/oracle/database-oracle.module.ts:42`), reached
unconditionally via `packages/database/src/sources/database-sources.module.ts:6`, in **both**
apps (`DatabaseModule` is `@Global`). No application service actually queries Oracle — the
boot dependency is pure wiring. The origin doc (§2.1, §9) sets the end state: no Oracle
datasource in the running app. The user's constraint: **stop using it, but do not delete the
files yet** (the generator + autogenerated entities stay useful for the one-time cutoff
migration, origin §5.2).

---

## Requirements

- R1. `@hsm/api` and `@hsm/worker` boot successfully with **no Oracle connection at runtime** and no Oracle host reachable — directly fixing the `ORA-12170` boot stall (origin §2.1, §9).
- R2. **All Oracle files are retained on disk**, not deleted — only unwired from the running app. The generator datasource (`packages/database/src/generator/datasources/oracle.datasource.ts`) and autogenerated Oracle entities stay intact for the future cutoff migration (user instruction; origin §5.2).
- R3. A **CI regression** guarantees the API boots Oracle-free — no `DB_ORACLE_*` env, no Oracle host — serving a pg-native endpoint (origin §8 "boot-without-Oracle" CI gate).
- R4. The **release train is landed and enforced**: workflows committed, `development` branch created, rulesets active, `pr-gate` the required check (origin §6–7).
- R5. A **living progress dashboard** tracks this plan's stage — each unit (U1–U5) and each roadmap wave (W0–W4) carries an explicit state, kept current as work lands (user request).

**Origin flows:** the request path reads Postgres; Oracle is never in it (origin invariant §9).

## Context & Research

### Relevant Code and Patterns

- **The eager boot seam:** `packages/database/src/sources/database-sources.module.ts:6` imports `DatabaseOracleModule`; that module's `TypeOrmModule.forRoot` (`.../oracle/database-oracle.module.ts:42`) is what dials Oracle at boot. Removing Oracle from this module's `imports`/`exports` is the single-point fix for both apps.
- **Config gate:** `packages/config/src/envs.ts:84-88` — `DB_ORACLE_HOST/USER/PASSWORD/DB` are `joi.string().required()`, `DB_ORACLE_PORT` defaults 1521. `envs.ts` throws at import if a required var is absent.
- **Pattern to preserve (do NOT touch):** `packages/database/src/sources/postgres/database-postgres.module.ts` — the Postgres `forRoot`/`forFeature` + `OnModuleInit` schema/migration logic. This is the datasource app services actually use (`@InjectRepository(Entity, DatabasesEnum.HsmDbPostgres)`).
- **Retained-but-unwired files:** `.../oracle/database-oracle.module.ts`, `.../oracle/database-oracle.entities.ts`, `packages/database/src/entities/autogenerated/oracle/*` (24 entities), `.../generator/datasources/oracle.datasource.ts` (CLI-only, already not in app boot). `DatabasesEnum.HsmDbOracle` (`.../sources/database-source.enum.ts:3`) stays.
- **Release-train artifacts (prepared on this branch, currently *uncommitted*; rulesets already applied to the repo):** `.github/workflows/{pr-validation,release-tag,back-merge}.yml`, `.github/rulesets/*.json`, `scripts/apply-github-rulesets.sh`, `.github/rulesets/README.md`.

### Institutional Learnings

- **`docs/solutions/test-failures/2026-05-06-nestjs-config-joi-validation-dotenv-conflict.md`** — `envs.ts` runs `import 'dotenv/config'` at load, so dotenv back-fills any key not already in `process.env` before Joi runs. Changing `DB_ORACLE_*` rules must be atomic with the test-setup shim and `.env.example`, or Jest surfaces confusing import-time Joi errors. Governs U2.
- **`docs/solutions/test-failures/2026-06-25-queue-barrel-import-breaks-config-mock-tdz.md`** — the runtime twin of ORA-12170: eager import-time env reads. If a guarded (not fully removed) datasource were ever needed, `forRootAsync` + `useFactory` is the deferral mechanism. Here we remove instead.
- **`docs/solutions/runtime-errors/2026-05-04-typeorm-entity-circular-import-silent-drop.md`** — editing `sources/` barrels risks silent entity drops that are **build-green but boot-broken**. `sources/postgres/index.ts` (and oracle equivalent) must export schemas only, never re-export `*.entities`. Verify both apps `start:dev` after barrel edits. Governs U1.
- **`docs/solutions/developer-experience/2026-06-25-dev-container-local-run-and-login.md`** — the local dev loop already runs infra-only (postgres/redis/minio, **no Oracle**); a `POST /v1/auth/login` → 200 is the "app booted" signal. Model the U3 CI assertion on this.

### External References

- None needed — the change is internal wiring against well-established local patterns.

## Key Technical Decisions

- **Hexagonal architecture (ports & adapters) — the app-wide style.** Domain/business logic
  sits at the center and depends only on **ports** (interfaces); everything external is a
  swappable **adapter**, so any provider — *or the database* — can change without touching
  logic:
  - **Driven (secondary) adapters:** persistence (Postgres via TypeORM behind a repository
    port — already the pattern), SRI e-invoice signing, payment gateway, email/SMS, Orthanc
    PACS, Panacea SQL Server (at cutoff), citizen lookup, biometric, WhatsApp.
  - **Driving (primary) adapter:** REST controllers — **one endpoint per capability** serving
    the internal app UI, external API clients, and the patient portal alike.
  - This reframes U1: **Oracle was a driven adapter; unwiring it is removing an adapter**,
    domain logic untouched. Same seam later hosts the cutoff-migration adapter (origin §5).
- **Unwire, don't delete (reconciles origin §9 with the user's "keep files").** The running app ends with *no Oracle datasource* (satisfies §9), achieved by removing `DatabaseOracleModule` from `database-sources.module.ts` — while the files remain on disk (satisfies R2). The end state and the retention are not in conflict: dead-but-present code is still "no datasource in the app."
- **Remove `DB_ORACLE_*` entirely from Joi + the `EnvVars` interface — not just optional.** The app no longer uses these, and the entity generator reads Oracle creds via its own CLI args (`packages/database/src/generator/utils/get_args.util.ts` `parseArgs`), **not** via `@hsm/config` — confirmed. So nothing in-process needs them; leaving them `optional()` would only keep live production Oracle creds loadable into the API/worker env for no benefit (needless blast radius). Drop them atomically with the test-setup shim and `.env.example`/`.env.template` (per the dotenv learning).
- **Boot-without-Oracle test is the gate for §9.** Once green, it's the signal to move `integration-tests` into `pr-gate`'s required `needs`.
- **Release train is verify/activate, not build.** Files exist and rulesets are applied this session; U4 commits them, creates `development`, and confirms enforcement — it does not re-author them.

- U1. **Unwire the Oracle datasource from app boot (keep files)**

**Goal:** Both apps stop connecting to Oracle at boot; all Oracle files remain on disk.

**Requirements:** R1, R2

**Dependencies:** None

**Files:**
- Modify: `packages/database/src/sources/database-sources.module.ts` (remove `DatabaseOracleModule` from `imports`/`exports`)
- Inspect/adjust barrels: `packages/database/src/sources/oracle/index.ts`, `packages/database/src/database.module.ts` (drop re-exports that force Oracle load)
- Retain untouched (on disk): `packages/database/src/sources/oracle/database-oracle.module.ts`, `.../oracle/database-oracle.entities.ts`, `packages/database/src/entities/autogenerated/oracle/*`, `packages/database/src/generator/datasources/oracle.datasource.ts`
- Test: covered by U3 (boot integration); plus verify no unit spec imports the Oracle connection

**Approach:**
- Remove the single eager entry: `DatabaseOracleModule` from `database-sources.module.ts`. Because nothing injects `@InjectDataSource(HsmDbOracle)` or the Oracle `forFeature` repositories outside `DatabaseOracleModule` itself, no service DI breaks.
- Keep `DatabasesEnum.HsmDbOracle` and all Oracle files importable but not registered.
- Watch the barrel trap: ensure `sources/oracle/index.ts` (if re-exported) doesn't drag `DatabaseOracleModule` back into the graph; export schemas/enums only.

**Technical design:** *(directional, not implementation spec)*
```
before:  DatabaseModule(@Global) → DatabaseSourcesModule → [DatabaseOracleModule(forRoot=EAGER dial), DatabasePostgresModule]
after:   DatabaseModule(@Global) → DatabaseSourcesModule → [DatabasePostgresModule]
         (DatabaseOracleModule + entities remain as files, simply not imported)
```

**Patterns to follow:**
- Mirror how `DatabasePostgresModule` stays the sole registered source.
- Barrel hygiene per `2026-05-04-typeorm-entity-circular-import-silent-drop.md`.

**Execution note:** After the edit, run `start:dev` for **both** `@hsm/api` and `@hsm/worker` — the circular-import/entity-drop failure mode is runtime-only and build-green.

**Test scenarios:**
- Integration (U3): API boots with no Oracle env and no Oracle host → success.
- Edge: grep confirms no app-layer `@InjectRepository(..., HsmDbOracle)` / `@InjectDataSource(HsmDbOracle)` remains active outside the retained (unimported) Oracle module.

**Verification:**
- `pnpm --filter @hsm/api start:dev` and `pnpm --filter @hsm/worker start:dev` both reach ready state ("Seeded default admin user") with Oracle unreachable.
- Oracle files still present in the tree (`git status` shows them unmodified/retained).

- U3. **CI regression: API boots Oracle-free**

**Goal:** Prove in CI that the API bootstraps and serves a pg-native endpoint with `LEGACY_ORACLE`/`DB_ORACLE_*` unset and no Oracle host — the guard for origin §9.

**Requirements:** R3

**Dependencies:** U1, U2

**Files:**
- Modify: `apps/backend/api/test/app.e2e-spec.ts` (assert boot + a pg-native endpoint responds; no Oracle env)
- Modify: `.github/workflows/pr-validation.yml` — the `integration-tests` job currently declares **only a `postgres` service** and omits `DB_REDIS_*` / `STRG_S3_*` env. It must **add `redis` + `minio` service containers** and the full dummy env (mirror `migration-drift.yml`, minus Oracle). Once green, add `integration-tests` to `pr-gate`'s `needs`.
- Test: `apps/backend/api/test/app.e2e-spec.ts` (and the worker e2e the job also invokes)

**Approach:**
- **"Boots without Oracle" ≠ "boots on Postgres alone."** Two facts, both verified: (1) `@hsm/config` marks `DB_REDIS_*` and `STRG_S3_*` as `required()` and throws at import if absent; (2) AppModule eagerly connects Redis (BullMQ `forRoot`) and **awaits `S3Initializer.ensureBuckets()`** against MinIO at init — the exact eager-network-at-boot shape Oracle had. So Redis + MinIO remain boot dependencies; only Oracle is intentionally absent. The job must provision `postgres` + `redis` + `minio` (the local-run infra set) with the full dummy env.
- **`test/jest-e2e.json` has no `setupFiles`**, so `test-setup.ts` does NOT run for e2e — the env must come from the CI job itself (or wire `setupFiles` into `jest-e2e.json`). Model the env on `migration-drift.yml`'s block; drop only `DB_ORACLE_*`.
- Assertion = local-run boot signal (`2026-06-25-dev-container-local-run-and-login.md`): ready + a pg-native route responds (e.g. `POST /v1/auth/login` → 200/401, not connection-refused).

**Execution note:** Start from a failing e2e that asserts boot without Oracle env, then make U1/U2 (and the CI service/env additions) satisfy it.

**Test scenarios:**
- Covers R3. Integration: app boots with postgres+redis+minio up, no `DB_ORACLE_*`, no Oracle host → a pg-native endpoint responds (not connection-refused).
- Error path: (documents the old failure) with the eager Oracle wiring this would have hung/`ORA-12170` — now it must not.

**Verification:**
- `pnpm --filter @hsm/api test:e2e` (and worker e2e) pass locally against postgres+redis+minio, no Oracle.
- `pr-validation` `integration-tests` job green on a PR (with redis+minio services added); `pr-gate` updated to require it.

## Module Build Order & Inventory (roadmap beyond this plan)

This foundation plan unblocks the module rebuilds. The **order** is dictated by the legacy
dependency graph (from the audits), not the generic R6–R37 taxonomy. Each wave/cluster below
becomes its own `/ce-brainstorm → /ce-plan → /ce-work` cycle; **cut-over happens per cluster**
at the end of its wave, only once the whole cluster is ported (origin §5).

### Dependency prerequisites (why the order is forced)

High fan-in legacy entities that almost everything reads/writes — so they must be rebuilt first:

| Legacy entity | Role | Consumed by |
|---|---|---|
| `CG_REF_CODES` | System-wide enum / reference-code dictionary | Nearly every module |
| `CG_CODE_CONTROLS` | App-managed numbering counters (concurrency-critical) | Invoices, docs, orders |
| `PERSONAL` | Staff master = app users (identity anchor) | RBAC, schedules, signatures, honoraria |
| `PACIENTES` | Patient master | Every HIS/HAS/SGI/Api module |
| `PERMANENCIAS_Y_ATENCIONES` | Admission/stay = **episode key** | All clinical notes, diagnoses, `CUENTAS` |
| `CUENTAS` | Patient account = charge bus | Billing, AR, SRI, insurer planillas |

Legacy chains that set sequencing: `PACIENTES → PERMANENCIAS → {evolución, diagnósticos, CUENTAS}`;
`CUENTAS → FACTURAS → {DETALLES, CxC, FACTURA_ELECTRONICA_SRI}`; `PERSONAL → privilegios → menú` (RBAC);
`PACIENTES → TURNOS_CE → HORARIOS_MEDICO → PERSONAL` (scheduling).

### API surface & frontend (not a backend wave)

There is **no separate "patient-facing" or "Api" backend wave** — because **the backend *is*
the API.** Per hexagonal design (see Key Technical Decisions):

- **Each module exposes REST endpoints as its public port; the backend runs with or without a
  frontend.** The Angular app is just one client of those endpoints — no different from an
  external integrator. There is no "API layer" to build.
- **"One endpoint" still needs per-caller authorization — token *type* ≠ authorization *scope*.**
  A staff session sees all patients (RBAC); a patient-portal caller must see **only their own
  rows** (ownership scoping); a partner API has its own scope. Every endpoint the patient
  portal consumes (e.g. `Resultados_Exámenes`, `AutoAdmisión`, `Pagos`) must enforce
  **row-level ownership at the domain/port layer** — "session vs API token" is not the whole
  access model. Treat this as a first-class requirement in each module's own `/ce-plan`.
- So the legacy `Api/*` apps are **just normal module endpoints**, living in the module that
  owns their data:
  - **Pagos** (`Api/PAGOS`) → **HAS billing** (W3.2), not patient-facing.
  - **AutoAdmisión** → the **Admisión** registration endpoint (W3.1).
  - **Resultados_Exámenes** → the **Laboratorio / Imágenes** results endpoint (W2.5 / W2.7).
- **Patient-facing is frontend.** The patient portal is an Angular surface calling those same
  module endpoints — no backend of its own. Its nav-frame already exists (roadmap).

**Cross-wave dependencies to respect:**
- **One owner for the episode entity.** The `PERMANENCIA`/Encounter entity is owned by the
  **W2.2 FHIR Encounter spine** (already partially built) — it is defined and migrated there,
  once. Admisión (W3.1) and every clinical note *populate/reference* it; they do not re-define
  it. This resolves the seam: W2.2 ships the entity + a stub open-episode path so it is
  testable before the full W3.1 Admisión workflow exists, and W3.1 later enriches it. (The
  subsystem numbering is a labelling convenience; the true order is dependency-cluster —
  shared services → Paciente → episode spine → clinical/billing.)
- **HIS clinical (W2) produces charges (`CUENTAS`)**; HAS billing (W3.2) consumes them — so
  HIS precedes HAS billing.
- **SGI (W4) writes into the clinical/billing schema** in legacy — the rebuild gives it clean
  boundaries onto the W2/W3 modules.

### Exclusions & notes

- **Dead / duplicate folders are NOT ported.** The audit flags backups (`- Copy`, `-RESPALDO`, `_dev`, numbered clones like `evolucion_enfermeria_1..13`), empty stubs (`ETIQUETAS_SANGRE`, `HAS/GAA/GTH`), and the full `Modulos/respaldo` mirror (~220 modules). Confirm live-vs-dead against `SN_CONTENIDO_MENU.NOMBRE_ARCHIVO` before excluding (audit open question).
- **Panacea (SQL Server) is a double migration** — folds into W3.4 accounting cut-over, plus citizen-lookup (`LISTA_HABITANTES`) and token service touch W2/W4 as needed (origin §5, Panacea note).
- **Scale (from audits):** ~7 subsystems → ~50+ modules → ~250+ Oracle tables, ~30+ PL/SQL procs, ~370 sub-app folders, ~25,000 PHP files; the billing engine (`Facturar`, 50+ files) and AR (`cuenta_cobrar`) carry the heaviest PL/SQL reimplementation.
- Waves are dependency-ordered, not rigid phases — later waves may start once their prerequisites (W1 shared services, W2.1 Paciente, and the W2.2/W3.1 episode-key spine) are ported.
