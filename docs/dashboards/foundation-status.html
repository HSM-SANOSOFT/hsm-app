architecture

Hexagonal — ports & adapters

Swap any provider — or the DB — without touching logic
Driving adapter = the API →
REST endpoints
The backend IS the API. Each module exposes REST endpoints — runs with or without a frontend. The Angular app is just one client, no different from an external caller.
Domain / business logic
depends only on ports (interfaces)
→ Driven adapters
Postgres / TypeORMSRI signingPayment gatewayEmail / SMSOrthanc PACSPanacea SQL Server (cutoff)Citizen lookupBiometricWhatsApp
Each hides behind a port — swapping one never edits domain code. The DB is an adapter too (already true via the ORM).
Corollaries: the backend is the API, so Api/* apps are just normal module endpoints living where their data does (Pagos → billing, AutoAdmisión → admissions, Resultados → lab) — nothing to "fold into a layer." Patient-facing is frontend (Angular calling those same endpoints), not a backend wave. Dropping Oracle = removing one driven adapter; domain untouched. Chatbot & signage: deferred (documented as extra clients of the same endpoints).

Why it works — the standalone model

pg-native, no runtime Oracle

build
Every module = full greenfield CRUD + unit + integration tests. Postgres is truth from day one.
parallel
Legacy stays the system of record with real users; new app runs as a standalone beta. No runtime link.
one writer
Legacy is the only production writer of a cluster until its cutover.

Cutoff — the one legacy touch

per cluster
Once a whole wave/cluster is ported: one-time bulk migration Oracle (+ Panacea) → Postgres.
prove
Reconcile + end-to-end on migrated data, then flip: new app becomes the writer.
GA
Repeat per cluster → legacy fully retired → v2 GA.
