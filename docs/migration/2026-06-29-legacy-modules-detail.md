# Legacy System Audit — Sub-App Inventory (deep pass)

> Companion to `2026-06-29-legacy-system-audit.md`. That file maps subsystems→modules; **this file drills one level deeper — every sub-app folder inside each module**, with its purpose, entry files, tables read/written (W = written), stored procs/sequences, and dead/duplicate flags.
> Source: SanoSoft legacy repo at `\\10.1.1.4\xampp\htdocs\Sanosoft`, commit `f38bb2f3`. READ-ONLY, source-only audit. Inferences marked INFERRED. No secret values reproduced.
> Method: 17 parallel read-only sub-app audits over the SMB mount. ~370 sub-app folders enumerated (asset/`node_modules`/`webfonts`/`lib` dirs excluded).

## How to read this

- **W** after a table name = that sub-app INSERTs/UPDATEs/DELETEs it; no suffix = read-only.
- **Procs/seqs** = Oracle PL/SQL called via `BEGIN proc(); END;` or `<SEQ>.NEXTVAL`. `CG_CODE_CONTROLS` is an app-managed counter table used as a sequence.
- Folders flagged **DEAD/DUP** are backups, dev copies, or superseded clones — confirm against the live menu (`SN_CONTENIDO_MENU.NOMBRE_ARCHIVO`) before migrating; do not port them.
- Tables here that are **not** in `2026-06-29-legacy-system-audit.md §4` are consolidated in §A below — they materially expand the migration table inventory.

# §B — Cross-cutting findings & dead-code summary

### Patterns confirmed at sub-app level
- **One canonical writer, many report clones.** Each module has 1–2 real transactional apps; the rest are read-only reports built from a shared `Comunes/lista.php` + `index_nuevo.php` + `export.php` + `envio_email.PHP` skeleton (most visible in COB COBRANZA and the SGI report families). The report skeleton frequently carries **copied `UPDATE CUENTAS` / `UPDATE TURNOS_CE` boilerplate** that is vestigial in a read-only report — verify it is unreachable, since a report mutating billing/turn state would be a latent bug.
- **`_TEM` draft/commit tables** in Emergencia (EMERGENCIAS_TEM, EXPLORACIONES_FISICAS_TEM, etc.) — a duplicate-table staging pattern that must be modeled in the new schema or replaced with a draft flag.
- **SGI/TIC write into the clinical & billing schema.** Management modules are NOT isolated: GSP/Brazalete, SGI/GTH & SSA certificate apps, SSA/ssa_ocupacional, SSA/actualiza_datos_colaboradores write `HOJAS_DE_EVOLUCION`, `DETALLES_DE_EXAMENES`, `ODONTOGRAMA`, `PACIENTES`, `PERSONAL`, `REGISTROS_PACIENTE`. Treat the whole Oracle schema as one DB for dual-write.
- **Destructive ops on live billing tables**: `TIC/SEG/produccion` and `FARMACIA/regulacion_botica`/`transferencias_producto` issue `DELETE FROM CUENTAS`/`CUENTAS_COPAGO`/`TRANSACCIONES` with concatenated `$valor` params — SQLi + data-loss risk.
- **Two ID-generation styles persist** across sub-apps: native `*_SEQ.NEXTVAL` and the `CG_CODE_CONTROLS` counter table (plus occasional `MAX+1/ROWNUM`). Unify atomically on migration.
- **Maintenance module mid-migration**: `MANTENIMIENTO` has a legacy `MSG_*` schema (ingreso_activos, orden_mantenimiento) being replaced by a new `MSGMNT_*` schema (`ingreso_activos_dev`, CRUD_ORDENES). Pick the `MSGMNT_*` target.
